# How to Write Agent Skills

This document is the authoritative standard for every skill file in this project.
Before writing a new skill or updating an existing one, read this in full.

---

## What a Skill Is

A skill is a structured markdown file that gives an agent explicit procedural knowledge
for a specific task or domain. It is injected into the agent's system prompt at runtime.

A skill is NOT:
- A data file or a knowledge base (that is what the FAISS vector store is for)
- A tool or a function the agent calls
- A replacement for the agent's core logic in Python
- A place to store session state or results

---

## File Format

Every skill file must begin with YAML frontmatter delimited by `---`, followed by
a markdown body. The frontmatter is machine-read. The body is injected into the agent.

### Required Frontmatter Fields

```yaml
---
name: <kebab-case-identifier>          # Unique within the agent's folder
description: <one sentence>            # What this skill does and when to apply it.
                                       # This is the most important field — it determines
                                       # whether the agent loads this skill.
agent: <agent-folder-name>             # e.g. architect, coder, researcher
version: <semver>                      # Start at 1.0, increment on meaningful changes
domains: [<domain>, ...]               # Applicable domains, or [all] if domain-agnostic
---
```

### Optional Frontmatter Fields

```yaml
source: <candidate-id or manual>       # If generated by Skills Writer, the candidate ID
score: <float>                         # Score of the candidate that produced this skill
generation: <int>                      # Generation number it came from
```

---

## Body Structure

The body must follow this section order. Omit sections that do not apply,
but never reorder them.

### 1. Purpose (required)
One short paragraph. What problem does this skill solve for the agent?
Why does the agent need explicit guidance here rather than its base training?

### 2. Key Principles (required)
3–7 bullet points. The non-negotiable rules the agent must follow when applying this skill.
These are the first things the agent reads, so they must be the most important things.

### 3. Procedure (required for procedural skills)
Numbered steps. Each step must be concrete and actionable.
Do not write steps like "consider the architecture" — write "select the base template
from the approved list below, then apply at most 2 mutations from the mutations list."

### 4. Domain-Specific Guidance (required for domain skills)
Subsections covering:
- Approved base templates for this domain (with brief rationale)
- Known-good mutation combinations
- Known dead ends (mutations to avoid)
- Common failure modes and their causes

### 5. Examples (optional but strongly encouraged)
Concrete examples of good and bad outputs. For architecture skills: good vs bad proposals.
For query skills: good vs bad arXiv queries. For rubric skills: good vs bad scores with reasoning.

### 6. Failure Mode Reference (required for validator/evaluator/critic skills)
A checklist of specific failure conditions the agent must check for.
Each item: condition, what it indicates, what the agent should output in response.

## Writing Rules

**Be specific, not general.**
Bad: "Use an appropriate attention mechanism."
Good: "Use multi-head self-attention with 4–8 heads. Do not use cross-attention unless
the proposal explicitly involves encoder-decoder architecture."

**Write for the agent, not for a human reader.**
The agent reads this as instructions. Use imperative language: "Select...", "Check...",
"Do not...", "Return...". Avoid passive voice and hedging.

**Keep the body under 500 lines.**
If a skill is growing beyond 500 lines, split domain-specific reference material into
a separate `references/` file and link to it. The agent reads the body in full on every call.

**Do not duplicate information already in the agent's Python code.**
Skills add knowledge the agent does not already have. Do not restate what the agent already
does by design (e.g. "call the LLM to generate a proposal" — the agent knows this).

**Version on meaningful change, not every edit.**
Increment the minor version for content improvements (1.0 -> 1.1).
Increment the major version if the structure or key principles change (1.1 -> 2.0).

## Template

Copy this when creating a new skill:

```markdown
---
name: <agent>-<domain>-<type>
description: <One sentence: what this does and when to apply it.>
agent: <agent-folder-name>
version: 1.0
domains: [<domain>]
---

# <Skill Title>

## Purpose

<One paragraph. Why does this agent need this skill?>

## Key Principles

- <Principle 1>
- <Principle 2>
- <Principle 3>

## Procedure

1. <Step 1>
2. <Step 2>
3. <Step 3>

## Domain-Specific Guidance

### Approved Base Templates

- **<Template name>** — <when to use it>

### Known-Good Mutations

- <Mutation> — <what it improves and under what conditions>

### Known Dead Ends

- <Mutation> — <why it fails in this domain>

### Common Failure Modes

- <Failure> — <cause> — <remedy>

## Examples

### Good Output
<example>

### Bad Output
<example and explanation of what is wrong>
```

---

## Quality Checklist

Before saving a skill, verify:

- [ ] Frontmatter is valid YAML and all required fields are present
- [ ] Description is one sentence and describes *when* to use this skill, not just *what* it is
- [ ] Body follows the section order above
- [ ] All procedures are written in imperative language with concrete steps
- [ ] No content duplicates what the agent's Python code already handles
- [ ] Body is under 500 lines
- [ ] Version is set correctly
- [ ] At least one example is included (for non-rubric skills)
